# 2026-06-26 Wework

![[2026-06-26-wework-whiteboard.png]]

## Objective
Turn the WeWork whiteboard and scratch notes into a grounded research plan for Keyframe's real-time persona stack, especially the split between S1 representation quality and S2 animation quality.

## Description
> [!note] Snapshot
> The meeting frames model progress as a two-track problem. S1 should improve the visual representation space: aspect ratio, resolution, body embeddings, framing, and decode quality. S2 should improve interactive animation: dyadic data, emotion conditioning, audio representations, chunking, video/context input, and post-training.

### Core Thesis
The whiteboard is not just asking for "better video." It is asking for a better representation boundary, carry meaning between raw inputs and final video. The repo already has a useful version of this boundary: Windy turns 1 second of audio plus a semantic prefix into semantic codes, then decodes those codes into frames. The research plan is to make that boundary carry more of what the product needs: body framing, emotion, turn-taking, gaze, screen context, posture, and richer dyadic timing.

### Whiteboard Synthesis
| Track                         | Meeting Idea                                                                                                                               | Repo Connection                                                                                                                         | Main Gap                                                                                                                |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------- |
| S1 representation             | Train body embeddings on native 16:9, 9:16, and 1:1 crops; explore foreground-to-frame at 288x512; improve resolution/decode quality.      | Windy Stage 1 maps semantic codes to images and can select 256 or 512 square output.                                                    | Current runtime and data paths still lean square: fixed `frame_size`, fixed frame tensors, and crop/resize assumptions. |
| S2 animation                  | Dyadic interaction data, mixed streams, cross-attention, WavLM/dmel/hierarchical representations, chunk-size experiments, and video input. | Windy Stage 2 already maps audio plus prefix plus optional emotion into semantic codes using cross-attention.                           | The repo has runtime turn/interruption traces, but no obvious first-class dyadic training-data schema.                  |
| Emotion                       | Add synthetic/calibrated data for emotions such as concerned and surprised.                                                                | Windy runtime maps `surprised` and `concerned`, while public SDK/plugin types expose only neutral, happy, sad, and angry.               | Capability discovery and client contracts need to catch up before new labels are product-safe.                          |
| Thinking state                | Continue separating small talk from heavier operations; consider hidden reasoning traces or smaller intermediate LLMs.                     | KFL voice-agent has STT to LLM to TTS tracing and explicit latency milestones.                                                          | Public avatar state is still mostly listening/speaking; "thinking" needs a product/runtime contract.                    |
| LoRA                          | Treat LoRA as a possible adaptation mechanism, not the whole strategy.                                                                     | InfiniteTalk has LoRA-loading code.                                                                                                     | LoRA can patch a model, but it does not solve missing calibrated representation data or missing interaction labels.     |
| Screen perception and posture | Add screen perception, posture, and possibly body-report LoRA as future controls.                                                          | There are product/demo surfaces for camera/screen-style interactions, but no clear avatar-runtime control path found in the first pass. | Decide whether these are model inputs, voice-agent context, or explicit control messages.                               |

### Data Strategy
The notes point to three data lanes:

| Data Lane | Proposed Source | Purpose | Notes |
| --- | --- | --- | --- |
| Pre-training representation data | Internet 16:9, Velvet 9:16, existing image/video corpora | Teach body/framing coverage before interaction-specific tuning. | Keep crop policy explicit instead of silently forcing 1:1. |
| Dyadic animation data | Velvet non-actor collection and KFL Studio | Learn timing, reaction, gaze, and turn-taking in paired interactions. | Vendor flywheel depends on better capture specs. |
| Post-training and calibrated data | KFL Studio actor tasks and labeled emotion data | Align model behavior to product-quality interaction moments. | Labels should include emotion, task state, gaze/screen posture, and perceived quality. |

### Suggested KFL Studio Collection
Use actor scenarios that force state changes, not just clean monologues:

- Actor has a computer problem and alternates between talking to camera, looking at the screen, typing, listening, and showing concern or surprise.
- Voice agent triggers small talk while a heavier operation continues in the background.
- Label the conditioning signal explicitly: emotion, gaze target, posture, screen-attention state, interruption, and whether the agent is thinking or speaking.
- Keep calibrated preference labels for "over deep" performance moments: naturalness, emotional appropriateness, turn timing, and visual continuity.

## Use Case
Use this note when deciding the next model/data sprint, writing data collection specs, or mapping research ideas into product/runtime changes.

### Action Plan
1. Define the S1 representation manifest:
   - Source aspect ratio: 16:9, 9:16, 1:1.
   - Framing transform: crop, letterbox, foreground-to-frame, or native.
   - Output target: 256 square, 512 square, 288x512 portrait, or another explicit shape.
   - Body/pose embedding availability.
   - Emotion and calibration labels, when present.

2. Run S1 ablations:
   - Current square-crop baseline.
   - Native aspect-ratio body embedding.
   - 512 latent output versus 256 output.
   - Foreground-to-frame portrait target.

3. Define the S2 dyadic capture spec:
   - User audio, assistant audio, transcript roles, turn boundaries, playback started/finished, interruption, emotion, gaze, screen-attention, posture, and quality labels.
   - Minimum clock-sync requirement for paired timelines.
   - Privacy rules for audio, screen, and transcript retention.

4. Align emotion contracts before expanding labels:
   - Decide whether `concerned` and `surprised` are internal-only model labels or public SDK labels.
   - Add model capability metadata so clients do not hard-code emotion vocabularies.
   - Add protocol contract tests across JS SDK, Elements adapters, KFL voice agent, LiveKit plugin, and Windy runtime.

5. Treat post-training as the last mile:
   - Try preference or RL-style methods only after the team has labeled examples of the target "v3 moment."
   - Keep GTDPO/RL as candidates, not default answers, until reward data and eval rubrics exist.

### Open Questions
- [ ] Where should representation datasets and calibrated labels live in the repo or adjacent data systems?
- [ ] Does S1 need rectangular shared-memory/runtime support, or can rectangular quality be handled before final runtime decode?
- [ ] Is "128 decode per frame" a concrete architecture target, a whiteboard shorthand, or a note about representation dimensionality?
- [ ] Should screen perception feed the voice agent, the avatar model, or both?
- [ ] Should "thinking" be a visible avatar state, an internal routing state, or a hidden trace state?
- [ ] What is the smallest dyadic dataset that can prove value before a larger Velvet/KFL Studio flywheel?

## Stage In Lifecycle
Research planning. This should become an experiment spec before implementation starts, then split into separate notes for data collection, runtime protocol changes, and model ablations.

## Importance
High. The note connects research direction to actual runtime constraints, which helps prevent abstract model plans from drifting away from the product system that must serve them.

## Risks
- Current Windy runtime assumptions are fixed around 1-second chunks, 25 frames, and square frame tensors. Rectangular or longer-context representations may touch more than training code.
- Emotion support is inconsistent across layers: the runtime knows more labels than public SDKs and plugins expose.
- LoRA exists in the repo, but relying on it too early could hide missing data and representation problems.
- Dyadic data can contain sensitive audio, transcripts, screen context, and behavioral labels.
- The whiteboard contains terms that are not yet mapped to source code, including Velvet specs, KFL Studio pipelines, GTDPO, "v3 moment," and "128 decode per frame."
